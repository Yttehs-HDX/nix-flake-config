# Architecture
## 总览
`nix-flake-config` 的目标，不是简单地组织一堆 Nix 模块，而是构建一套稳定的配置架构，使多用户、多主机、多后端可以在同一套声明体系下被统一描述、组合与实现。

这个架构的核心思想是：
- 用户配置独立存在
- 主机配置独立存在
- 用户配置优先表达“我要什么”
- 主机配置只表达“这台机器是什么、能做什么”
- 用户与主机之间的关系，不由任意一方单独决定，而由顶层关系网统一维护
- 后端根据关系网与目标主机类型，将声明精确落地到正确的位置

换句话说，这个项目不是“谁 import 谁”的问题，而是“谁声明什么、谁属于哪里、最终如何实现”的问题。

---
## 设计目标
本架构希望满足以下目标：

### 1. 支持多用户与多主机的任意组合
同一个用户可以出现在多台主机上。  
同一台主机也可以承载多个用户。  
这些组合不应依赖临时拼接，而应由明确的模型表达。

### 2. 用户配置独立于主机
用户配置不应天然依附于某台主机。  
一个用户的配置应当能够同时服务于不同类型的主机后端，例如：
- home-manager
- nixos
- nix-darwin

用户配置首先描述“这个用户需要什么”，而不是“这个用户在某台机器上怎么写”。

### 3. 主机配置不能反向定义用户意图
主机可以声明自己的能力、平台与限制，但不应该直接改写用户配置本身。

也就是说，主机可以决定：
- 自己是什么类型
- 自己支持哪些能力
- 自己承载哪些用户
- 某些只能属于主机的系统级实现

但主机不应该直接说：
- 某个用户突然多了什么偏好
- 某个用户的核心声明被重写
- 某个用户的抽象身份被主机任意篡改

### 4. 配置必须精确落地
用户的声明最终不能只是停留在抽象层。  
它必须被准确翻译到具体目标中，并且具备明确作用域。

例如：
- 用户组配置，必须落地到某台主机上的某个具体用户
- 用户软件配置，必须只影响该用户自己的 home 层
- 主机服务配置，必须只作用于该主机自己的 system 层
- 某些声明可能跨后端展开，但最终每一份结果都必须可定位、可解释、可追踪

---
## 核心对象
本架构将系统拆分为四类核心对象：

### 1. User
`User` 表示用户级声明单元。

它描述的是一个“用户希望拥有的配置能力与行为”，例如：
- 用户偏好
- 用户软件集合
- 用户桌面能力
- 用户主题声明
- 用户开发环境
- 用户级服务
- 用户希望参与哪些主机关系

`User` 本身不直接等于某个后端实现，也不直接等于某台机器上的某个账户结果。  
它是一个抽象声明对象。

---
### 2. Host
`Host` 表示主机级声明单元。

它描述的是一台主机自身的事实与能力，例如：
- 主机类型
- 平台信息
- 系统级能力
- 硬件相关配置
- 网络、引导、服务等系统级行为
- 该主机是否具备桌面环境
- 该主机是否允许承载某类用户能力

主机只负责定义“环境”，不负责定义“用户意图”。
---

### 3. Backend
`Backend` 表示主机的实现后端。

当前已知的后端类型有：
- `home-manager`
- `nixos`
- `nix-darwin`

其中：
- `home-manager` 表示纯用户层主机
- `nix-darwin` 表示 Darwin 系统主机
- `nixos` 是一种特殊主机，天然同时拥有：
  - `nixos` system 部分
  - `home-manager` user 部分

因此，`NixOS` 主机不是单一层，而是双层实现目标。

---

### 4. Relation Graph
`Relation Graph` 表示用户与主机之间的关系网。

这是本架构中的关键拓扑层。  
它独立于用户配置与主机配置本身，用于描述：
- 哪些用户存在于哪些主机上
- 某个用户在某台主机上的身份映射
- 某个关系是否启用
- 某些与“用户-主机组合”相关的附加参数
- 用户声明在某台主机上的实例化上下文

这意味着：
- 不是用户单方面选择主机
- 也不是主机单方面选择用户
- 而是由一个独立的顶层关系层，显式描述两者之间的连接

这样可以避免模型失真，也更容易维护复杂拓扑。
---

## 为什么需要独立的关系网
如果让用户直接选择主机，会导致用户配置携带过多部署拓扑信息。  
这样用户声明就不再纯粹，而会逐渐变成“带目标机器列表的半实现配置”。

如果让主机直接选择用户，则主机会天然拥有更强的控制权，容易反向影响用户配置语义。  
这会破坏“用户配置为主”的原则。

因此，更稳定的做法是：
- 用户只定义自己
- 主机只定义自己
- 顶层额外定义二者之间的关系

这个关系层可以被视为系统的“部署图”或“拓扑图”。

它的职责不是发明配置，而是决定：
> 哪些声明，在什么上下文中，被投影到哪些目标上。

---

## 配置分层
整个架构可以分成三层：

### 1. 声明层
声明层用于表达对象本身的意图与属性，包括：
- 用户声明
- 主机声明

这一层不关心具体如何 import，不关心具体落在哪个 backend，也不直接关心某个模块文件路径。

它只表达“是什么”和“想要什么”。

---
### 2. 关系层
关系层用于表达对象之间的拓扑关系，包括：
- 用户与主机的绑定关系
- 用户在主机上的实例化信息
- 用户是否启用在某台主机上
- 某些组合态参数

这一层是整个系统的编排中心。
---

### 3. 实现层
实现层负责把声明层与关系层的信息，翻译为真正的后端配置，包括：
- NixOS system 配置
- Home Manager user 配置
- nix-darwin 配置
- 软件包行为适配
- 服务生成
- 用户组与权限映射
- 平台差异处理

这一层的职责是“落地”，不是“重新定义需求”。
---

## 基本原则
### 原则一：用户配置为主
用户配置是整个系统中最主要的需求表达入口。

这并不意味着所有配置都属于用户，  
而是意味着：
- 与个人偏好、个人软件、个人桌面体验、个人开发环境相关的内容，应优先属于用户配置
- 用户配置应作为主要声明源，被后端解释与实现
- 用户不需要为了每种主机后端重新写一套配置

---
### 原则二：主机配置不能影响用户配置本体
主机配置不能直接修改用户配置定义本身。

主机可以提供上下文，例如：
- 当前平台是 Linux 还是 Darwin
- 当前 backend 是 home-manager 还是 nixos
- 当前机器支持哪些系统能力
- 当前机器是否允许某些系统级实现

但这些上下文只应影响“如何实现”，而不应影响“用户本来想要什么”。

也就是说，主机只能影响用户声明的**投影结果**，不能改写用户声明的**原始语义**。

---
### 原则三：跨主机困难的配置，应视为主机配置
如果一项配置天然难以跨主机共享，或者高度依赖机器事实，那么它就不应属于用户配置，而应属于主机配置。

例如：
- 硬件驱动
- 文件系统挂载
- 引导项
- 网络接口
- 内核参数
- 机器特定服务
- 强依赖平台差异的实现

判断标准不是“它看起来像系统配置”，而是：
> 这项声明能否在不同主机类型之间保持稳定语义？

如果不能，就更适合归入主机配置。
---

### 原则四：同一份用户配置服务所有 backend
用户配置必须尽可能以统一抽象服务不同 backend：
- 对于 `home-manager` 主机，主要落地到用户层
- 对于 `nix-darwin` 主机，根据 Darwin 能力进行适配
- 对于 `nixos` 主机，可能同时影响：
  - system 侧必要的系统实现
  - home-manager 侧用户实现

这要求用户配置本身不要与某个 backend 强耦合。  
backend 差异应主要由实现层吸收。

---
### 原则五：所有落地都必须具备唯一作用域
配置落地时必须明确回答三个问题：
- 作用于谁
- 作用于哪台主机
- 作用于哪个后端层

例如用户组配置：
- 不能是“所有用户共享一份组配置”
- 而必须是“某台主机上的某个用户，需要加入哪些组”

也就是说，系统必须能够识别出：
- 当前用户是谁
- 当前主机是谁
- 当前是主机 system 落地，还是用户 home 落地

只有这样，配置才是准确且唯一的。

---
## NixOS 主机的特殊性
`NixOS` 主机是一种特殊主机，因为它同时具有两种实现面：

### 1. System 面
用于承载系统级配置，例如：
- 系统服务
- 用户账户定义
- 用户组
- system packages
- 网络与引导
- 桌面管理器
- 系统级桌面支持

### 2. Home 面
用于承载用户级配置，例如：
- 用户软件
- Shell 配置
- 编辑器配置
- 用户服务
- 主题与桌面偏好
- 用户侧开发环境

因此，当一个用户被绑定到某台 `NixOS` 主机时，系统可能需要将同一份用户声明拆分成两个部分：
- 一部分投影到 system
- 一部分投影到 home-manager

这种双重投影，是 `NixOS` 主机最重要的特殊点。

---
## 精确落地模型
为了实现“用户声明 -> 后端实现”的自动化，系统需要支持一种精确落地模型。

这个模型至少包含以下维度：
- `subject`：当前声明属于谁
- `host`：当前目标主机是谁
- `backend`：当前目标后端是什么
- `scope`：当前是 system 还是 home
- `instance`：当前是不是某个用户在某台主机上的实例

一个声明是否被启用，不应只看“全局开关”，而应看它是否匹配当前实例上下文。

例如：
- 某个用户的软件配置，只应在该用户出现在某台主机上时生效
- 某个用户组映射，只应在 NixOS system 层展开
- 某个桌面主题，只应在支持桌面的目标上落地
- 某个 Darwin 不支持的实现，应自动跳过或改走兼容路径

---
## 模块与软件包如何读取 config
系统允许模块或软件包根据全局 `config` 读取上下文信息，从而实现适配行为。  
这是实现自动化落地的重要能力。

例如，一个软件包模块可以根据：
- 当前用户声明
- 当前主机类型
- 当前桌面能力
- 当前主题风格
- 当前 backend 能力

来决定自己的行为。

但是这里需要一个重要边界：
> 允许读取 config，不等于鼓励无边界耦合。

如果所有模块都随意读取任意路径的 config，系统最终会重新退化为隐式依赖网络。  
因此更合理的原则是：
- 模块可以读取 config
- 但应优先读取稳定、公开、可预期的配置接口
- 尽量避免直接依赖别的模块的内部实现细节
- 被跨模块读取的配置，应被视为公共能力接口，而不是私有临时字段

也就是说，架构允许“全局感知”，但不鼓励“无边界串线”。

---
## 推荐的拓扑模型
当前最推荐的模型是：

### 1. 用户配置独立维护
描述用户本身的长期能力与偏好。

### 2. 主机配置独立维护
描述主机本身的类型、平台与系统能力。

### 3. 顶层维护关系网
显式记录：
- 哪些用户部署到哪些主机
- 某用户在某主机上的用户名、组、附加参数
- 某主机是否启用某用户
- 某关系是否需要特殊适配

### 4. 后端根据关系自动投影
根据目标主机的 backend 类型，将声明拆解并落地到：
- NixOS system
- Home Manager
- nix-darwin
- 其他未来 backend

---
## 结果形态
理想情况下，整个系统应形成这样的工作流：
1. 用户声明自己的能力与偏好
2. 主机声明自己的类型与能力
3. 顶层关系网定义用户与主机的连接关系
4. 后端读取当前实例上下文
5. 系统将声明自动投影到对应 backend
6. 每一项配置都在唯一且正确的位置落地

最终，使用者面对的不是混乱的 import 网络，而是一个稳定的配置系统：
- 用户是用户
- 主机是主机
- 关系是关系
- 实现是实现

每一层各司其职，不互相越权。

---
## 架构边界
本架构当前不追求：
- 让所有声明都完全脱离 backend 差异
- 把所有系统实现细节隐藏到不可理解
- 允许主机任意重写用户语义
- 用全局 config 任意穿透所有模块边界
- 为了抽象统一而牺牲可维护性

本架构真正追求的是：
- 稳定的对象边界
- 清晰的关系拓扑
- 自动化但可解释的配置落地
- 长期可扩展的系统结构

---
## 一句话概括
`nix-flake-config` 的架构，是一种以用户声明为主、以主机能力为环境、以关系网为拓扑、以后端投影为落地方式的配置系统。

它试图把复杂的多用户、多主机、多后端组合，从“手工拼接模块”，提升为“结构清晰、作用域明确、实现可追踪”的声明式架构。
