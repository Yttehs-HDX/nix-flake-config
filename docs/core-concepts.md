# Core Concepts
## 目标
本文档定义 `nix-flake-config` 中最核心的一组概念。  
这些概念用于统一描述系统中的对象、关系、作用域与落地方式。

本项目不是简单地组织模块文件，而是在构建一套声明式配置系统。  
因此，必须先明确：
- 系统中有哪些对象
- 每个对象分别表达什么
- 对象之间如何建立关系
- 一项声明最终如何被实现到具体后端

如果这些概念不稳定，后续的目录设计、模块边界、参数传递、配置投影都会逐渐失去一致性。

---
## 设计原则
本文档中的术语定义遵循以下原则：

### 1. 先定义语义，再定义实现
术语首先描述“它是什么”，而不是“它写在哪个文件里”。

### 2. 一个词尽量只表达一个核心含义
避免一个概念在不同上下文中拥有完全不同的解释。

### 3. 术语应服务于架构边界
好的术语不仅能帮助表达，还能帮助限制越界行为。

### 4. 概念优先保持稳定
实现可以演化，但核心概念不应频繁改变。

---
## 1. Declaration
### 定义
`Declaration` 表示一项配置声明。  
它描述的是一个对象“想要什么”或“具备什么”，而不是“如何实现”。

### 说明
声明是本项目最基础的输入单位。  
整个系统的工作，就是把声明逐步解释、约束、投影，并最终实现到具体后端。

声明不等于模块文件，也不等于最终生成结果。  
它更接近一种结构化意图。

### 例子
- 某个用户需要 `git`
- 某个用户启用 `hyprland`
- 某台主机使用 `nixos`
- 某台主机启用图形桌面
- 某个用户在某台主机上加入某个用户组

这些都可以视为声明。

---
## 2. Subject
### 定义
`Subject` 表示声明所属的主体。  
也就是：**这项声明究竟属于谁**。

### 说明
任何配置在进入系统时，都应尽量回答“归属”问题。  
如果不能明确归属，后续就很难判断它应该如何组合与落地。

在本项目中，主体通常包括：
- `User`
- `Host`
- `Relation`

### 例子
- “用户安装 neovim” 的主体是 `User`
- “主机启用 NetworkManager” 的主体是 `Host`
- “某用户在某台 NixOS 主机上加入 wheel 组” 的主体是 `Relation`

---
## 3. User
### 定义
`User` 表示用户级主体。  
它描述一个用户长期稳定的意图、偏好与能力需求。

### User 的职责
`User` 应主要表达：
- 用户软件偏好
- 用户桌面偏好
- 用户 shell / editor 配置
- 用户主题偏好
- 用户开发环境
- 用户级服务
- 与个人使用习惯强相关的能力声明

### User 不应直接表达的内容
`User` 不应直接表达高度依赖具体主机事实的配置，例如：
- 某块硬件驱动
- 某台机器的引导器
- 某个磁盘挂载点
- 某个网卡名称
- 某台机器专属的系统服务

这些内容更适合属于 `Host`。

### 判断标准
如果一项配置能在不同主机之间保持大体稳定的语义，它更可能属于 `User`。  
如果它严重依赖某台机器本身，它更可能不属于 `User`。

---
## 4. Host
### 定义
`Host` 表示主机级主体。  
它描述一台主机自身的事实、能力与系统环境。

### Host 的职责
`Host` 应主要表达：
- 主机类型
- 系统平台
- 硬件相关配置
- 网络与引导配置
- 系统服务
- 文件系统
- 桌面环境承载能力
- 主机级安全策略
- 机器本身具备或不具备的系统能力

### Host 不应直接表达的内容
`Host` 不应直接重写用户本体的偏好。  
它可以提供实现上下文，但不应直接篡改用户声明的原始语义。

例如：
- Host 可以说“这是一台 Darwin 主机”
- Host 可以说“这台机器没有图形界面”
- Host 可以说“这台机器允许/不允许某些系统实现”

但 Host 不应直接说：
- 某个用户突然改用另一套主题
- 某个用户的软件偏好被主机覆盖
- 某个用户的抽象能力定义被重写

---
## 5. Backend
### 定义
`Backend` 表示配置最终落地的实现后端。

### 当前已知 Backend
- `home-manager`
- `nixos`
- `nix-darwin`

### 说明
Backend 不是主体。  
它不回答“这项配置属于谁”，只回答“这项配置最终由谁实现”。

同一个用户声明，在不同 backend 下可能对应不同实现方式。

### 特别说明：NixOS
`nixos` 是一种特殊 backend，因为它通常包含两个实现面：
- system 面
- home-manager 面

因此，在 `NixOS` 主机上，一项用户声明可能会被拆分并分别落地到：
- `nixos` system
- `home-manager`

---

## 6. Scope
### 定义
`Scope` 表示声明最终生效的作用域。  
它回答的是：**这项配置落在哪一层**。

### 当前核心 Scope
- `system`
- `home`

### 说明
`Scope` 与 `Backend` 相关，但不完全相同。

例如：
- 在 `home-manager` 中，通常只有 `home` scope
- 在 `nix-darwin` 中，可能存在偏系统侧的 scope
- 在 `nixos` 中，同时存在 `system` 与 `home`

### 例子
- 用户 shell 配置通常落在 `home`
- 用户组配置通常落在 `system`
- 主机网络配置通常落在 `system`

---
## 7. Relation
### 定义
`Relation` 表示 `User` 与 `Host` 之间的绑定关系。  
它描述的是：**哪个用户以什么方式出现在什么主机上**。

### 为什么需要 Relation
如果没有 `Relation`，系统就会被迫在两种不理想模型中二选一：
- 用户直接选择主机
- 主机直接选择用户

这两种方式都容易让对象职责混乱。

因此，本项目将绑定关系提升为独立对象。  
这意味着：
- `User` 只负责定义自己
- `Host` 只负责定义自己
- `Relation` 负责定义二者如何连接

### Relation 应表达的内容
- 某个用户是否出现在某台主机上
- 该用户在该主机上的实例是否启用
- 该用户在该主机上的用户名映射
- 某些仅对“用户-主机组合”成立的附加参数
- 某些必须依赖具体部署上下文的配置

### 例子
- 用户 `alice` 出现在主机 `laptop`
- 用户 `alice` 在 `laptop` 上的实际用户名是 `alice`
- 用户 `alice` 在 `server-a` 上不启用桌面
- 用户 `bob` 在某台 NixOS 主机上加入 `wheel` 组

这些都更适合作为 `Relation` 信息，而不是纯 `User` 或纯 `Host` 信息。

---
## 8. Instance
### 定义
`Instance` 表示某个主体在具体上下文中的实例化结果。  
在当前架构下，最重要的实例通常是：
> 某个用户在某台主机上的具体实例

### 说明
`User` 是抽象声明对象。  
但真正落地时，系统需要处理的不是抽象用户，而是：
- 这个用户
- 在这台主机上
- 面向这个 backend
- 处于这个 scope
- 以这个名字和参数进行实现

这就是 `Instance`。

### 例子
- `alice@laptop`
- `alice@workstation`
- `bob@server-a`

虽然它们可能都来自同一个用户定义，但它们是不同实例。

---
## 9. Projection
### 定义
`Projection` 表示将声明从抽象语义映射到具体目标的过程。  
可以理解为“投影”或“展开”。

### 说明
这是本项目中非常关键的概念。

用户或主机的声明本身通常是抽象的，  
系统需要根据：
- 当前主体
- 当前主机
- 当前 relation
- 当前 backend
- 当前 scope
- 当前平台能力

把这份声明转换为具体实现结果。

这个转换过程就是 `Projection`。

### 例子
一个用户声明：
- 启用桌面终端
- 使用某个主题
- 加入某个用户组

在不同主机上可能被投影为：
- `home.packages`
- `programs.<name>.enable`
- `users.users.<name>.extraGroups`

这些都属于投影后的结果。

---
## 10. Realization
### 定义
`Realization` 表示声明经过投影后，最终形成的具体实现结果。

### 区别
- `Declaration`：我要什么
- `Projection`：系统如何把“我要什么”映射到目标
- `Realization`：最终生成了什么

### 说明
`Realization` 是最接近 backend 输出的一层。  
它通常已经不再是抽象概念，而是具体 Nix 配置片段或模块结果。

---
## 11. Capability
### 定义
`Capability` 表示某个对象具备的一项能力或支持的一类特性。

### 说明
Capability 不是具体实现，而是一种可被依赖、可被检查的语义能力。

它可以属于 `Host`，也可以属于 `User`，但两者含义不同。

### Host Capability 例子
- 支持图形桌面
- 支持 system 层用户管理
- 支持 Darwin 特性
- 支持 Wayland
- 支持某类服务运行环境

### User Capability 例子
- 需要开发环境
- 需要桌面体验
- 需要终端工具链
- 需要主题系统

### 为什么需要 Capability
因为很多投影逻辑不应直接依赖“某个具体模块路径”，  
而应依赖“当前环境是否具备某种能力”。

这样系统会更稳定。

---
## 12. Context
### 定义
`Context` 表示当前进行配置投影时可用的一组上下文信息。

### Context 可能包含
- 当前 user
- 当前 host
- 当前 relation
- 当前 backend
- 当前 scope
- 当前平台
- 当前 capability 集合
- 当前实例是否启用某类功能

### 说明
很多模块的行为都依赖 context。  
例如一个软件包模块可能需要知道：
- 当前是不是桌面环境
- 当前是不是 Darwin
- 当前用户是否启用了某项能力
- 当前主机是否支持 system 侧实现

这些信息共同组成 context。

---
## 13. Public Config Interface
### 定义
`Public Config Interface` 表示允许其他模块稳定读取的公共配置接口。

### 说明
本项目允许模块读取 `config` 来进行适配。  
但如果所有模块都随意读取彼此内部字段，系统最终会退化为隐式耦合网络。

因此需要区分：
- 公共接口
- 私有实现细节

### 原则
一个字段如果需要被多个模块稳定依赖，就应该被视为公共接口。  
否则，它更应该被保留为局部实现细节。

### 目标
允许“全局感知”，但避免“无边界串线”。

---
## 14. Ownership
### 定义
`Ownership` 表示一项声明的归属权。  
它回答的是：**谁有权定义这项配置**。

### 说明
在本项目中，归属权比“写在哪里”更重要。

因为同一项结果可能落到多个 backend，  
但它的语义归属仍然只能属于一个核心主体。

### 归属示例
- 用户偏好归 `User`
- 主机硬件归 `Host`
- 用户在某主机上的用户名映射归 `Relation`
- 某个 backend 的具体生成细节归实现层，而不是前端声明层

---
## 15. Invariant
### 定义
`Invariant` 表示系统中不应被破坏的核心不变量。

### 当前建议不变量
#### 1. 用户配置独立于主机定义
用户配置可以服务多个主机，不应天然绑定某一台机器。

#### 2. 主机不能直接改写用户本体语义
主机只能提供上下文和实现限制，不能重写用户的核心意图。

#### 3. 用户与主机的绑定关系独立存在
绑定关系由 `Relation` 表达，而不是由 `User` 或 `Host` 单方面吞并。

#### 4. 所有落地必须具备唯一作用域
每项实现都应能回答：
- 属于谁
- 作用于哪台主机
- 落在哪个 backend
- 落在哪个 scope

#### 5. 跨主机难以共享的配置不应强行属于 User
如果语义无法稳定跨平台，就应重新评估归属。

这些不变量会直接影响后续目录结构、模块边界与 option 设计。

---
## 核心对象关系图（概念层）
可以用下面这组关系理解整体架构：
```text
User ----\
          \
           > Relation ----> Instance ----> Projection ----> Realization
          /
Host ----/

Host ----> Backend
Backend -> Scope
Projection uses Context
Context includes User / Host / Relation / Backend / Scope / Capability
```

这张图表达的是：
- `User` 与 `Host` 不直接吞并彼此
- 二者通过 `Relation` 建立连接
- 连接后形成具体 `Instance`
- `Instance` 在具体 `Context` 中被 `Projection`
- 最终得到 `Realization`

## 一个简单例子
假设存在：
- 用户 `alice`
- 主机 `laptop`
- `laptop` 是 `NixOS`
- `alice` 想要 `git`、`zsh`、图形终端
- `alice` 在 `laptop` 上需要加入 `wheel` 组

那么可以这样理解：

### Declaration
- `alice` 需要 `git`
- `alice` 需要 `zsh`
- `alice` 需要桌面终端
- `alice@laptop` 需要加入 `wheel`

### Subject
- 前三个声明属于 `User`
- 最后一个声明属于 `Relation`

### Backend
- `laptop` 使用 `nixos`
- 同时具备 `system` 与 `home` 两个 `scope`

### Projection
系统根据上下文决定：
- `git` -> `home.packages`
- `zsh` -> 用户 shell 配置
- 桌面终端 -> 图形环境相关 home 配置
- `wheel` -> `users.users.alice.extraGroups`

### Realization
最终生成具体 NixOS + Home Manager 配置结果。

## 术语使用规范
后续文档建议遵循以下规范：

### 1. 不要把 User 与 Instance 混用
“用户”是抽象对象。
“某用户在某主机上的实例”是具体部署对象。
这两者不能混为一谈。

### 2. 不要把 Backend 与 Scope 混用
`Backend` 是实现后端。
`Scope` 是落地层次。
两者有关联，但不是同一个概念。

### 3. 不要把 Relation 当成临时参数堆
`Relation` 是架构中的正式对象，
不是“顺手补一个 hostUsers = { ... }”的临时容器。

### 4. 不要让 Host 直接拥有 User 的全部定义权
如果 Host 能完全定义 User，
那么 User 就失去了独立抽象的意义。

### 5. 不要让 config 读取变成隐式依赖泥潭
跨模块读取应优先依赖公共接口，而不是内部字段。

---
## 一句话总结
`nix-flake-config` 的核心概念体系可以概括为：
- 用 `User` 和 `Host` 表达独立主体
- 用 `Relation` 表达二者绑定关系
- 用 `Instance` 表达具体部署对象
- 用 `Backend` 与 `Scope` 表达实现目标
- 用 `Projection` 把声明映射为实现
- 用 `Realization` 得到最终配置结果

这套概念的目的，是让系统始终知道：
> 一项配置属于谁，在哪个上下文中生效，并最终如何准确落地。
